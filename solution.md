## 1. Описание решений (до 1 страницы)

### Задача 1 — Парсер 2ГИС

**Какой подход выбрали и почему**

Мы использовали **Playwright + перехват сетевых ответов (network interception)**.

Логика выбора:

Первым делом попробовали прямые запросы к `catalog.api.2gis.com` через `requests`. Получили 403 — API требует сессионные куки и API-ключ, привязанный к домену `2gis.kz`. Ключ меняется, реплей заголовков живёт минуты.

Второй вариант — DOM-скрапинг через Playwright (парсить HTML карточек). Проблема: 2GIS рендерит карточки через React с динамическими CSS-классами, которые меняются при каждом деплое. Хрупко.

Третий вариант — тот же Playwright, но с перехватом **JSON-ответов**, которые браузер получает от API. Браузер авторизован, куки есть, API отдаёт данные — мы их просто читаем на лету через `page.on("response", ...)`. Это устойчиво и даёт структурированные данные без парсинга HTML.

**Почему по районам?**

Поисковый эндпоинт 2GIS пагинирует и обрезает выдачу примерно на 100 объектах при общем запросе. Разбивка по 8 административным районам Алматы позволила получить 800+ уникальных записей вместо ~100.

**Data engineering (clean_data.py)**

Сырые данные содержат:
- Пустые поля `paid`, `district`, `parking_type` — заполнили эвристиками по названию и адресу
- Координаты иногда в формате `point.lat/lon`, иногда в `lat/lon` — унифицировали
- Дубликаты по `id` — удалили
- Добавили `google_maps_url` для удобства

**Результат:** 818 уникальных парковок, все поля заполнены насколько позволяют данные 2ГИС.

### Задача 2 — WhatsApp Sender

Использовали **Playwright + WhatsApp Web**. Это тот же подход, что используют популярные библиотеки типа `whatsapp-web.js` — открываем браузер, сканируем QR один раз, сессия сохраняется локально.

`dry-run` — режим по умолчанию: все сообщения пишутся в `dry_run_messages.log`. Запускали на полной базе из задачи 1 — лог содержит 818 записей.

`live` — запускается с флагом `--live --limit 2`, отправляет на собственный номер. Скриншоты/скринкаст прилагаются.

Обработка ошибок: невалидные номера (`normalize_phone`) — `skipped`; таймаут соединения — `failed` с описанием; программа не падает, продолжает следующую запись.
